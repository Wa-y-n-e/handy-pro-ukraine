# Profile Settings Beta Hub

Implementation date: 2026-07-01

## 1. What was added

- A self-only Settings gear in the existing Profile header.
- A dedicated mobile-first `/profile/settings` route.
- Account email, authenticated role, and an honest beta notice.
- Sign out with a busy state and visible error handling.
- Confirmed account-deletion requests through `support_requests`.
- A validated local language preference.
- A persisted simplified-mode preference shared with Home.
- Short Ukrainian terms, privacy, and beta-test disclosures.
- A typed beta problem-report form.
- Self-only master setup guidance and a debt-block warning without displaying the balance.

- `src/routes/profile.tsx`

## 3. Migrations

No migrations were added. The implementation uses the existing `support_requests` table and the
generated contract restored by PR #6.

## 4. Supabase tables and RPCs used

The Settings screen directly inserts into `support_requests` with its generated `Insert` type. It
does not call admin APIs, service-role clients, destructive profile operations, or new RPCs.

The existing shared session flow still reads `user_roles` and calls `get_my_profile` to provide the
authenticated role and private owner profile. No RPC definition was added or changed.

## 5. Settings route and entry

The Profile header shows a Lucide Settings icon only when `isSelf` is true. Public views opened as
`/profile?id=<other-user>` do not render it. The icon opens `/profile/settings`, which is registered
as an independent root child so the existing Profile page does not need to become a layout.

The route is behind the existing `AuthGate`, includes a back button to `/profile`, and keeps the
existing five-tab bottom navigation. No sixth tab was added.

## 6. Sign out

The old Profile footer action was replaced by one Settings action. It calls the existing
`supabase.auth.signOut()` flow, disables itself while running, and shows an error toast if sign out
fails. On success, the existing auth listener and `AuthGate` redirect to `/auth`.

## 7. Delete-account request

The client never deletes the Auth user, profile, or operational data. After an explicit confirmation
the screen inserts this typed support request:

- `created_by`: current authenticated user id;
- `kind`: `callback`;
- `duration_seconds`: `null`;
- `note`: `[delete_account] Користувач просить видалити акаунт.`

The UI says clearly that this is a support request and that account/data deletion is not immediate.
Success and failure are reported with toasts.

## 8. Language preference

The validated values `ua`, `ru`, and `en` are stored under `handy-pro.language`. Ukrainian is the
SSR and invalid-value fallback. Russian and English are labelled as coming soon, and the screen
states that the interface remains Ukrainian because full i18n is postponed.

No profile fields or sensitive data are stored with this preference.

## 9. Simplified mode

The boolean preference is stored under `handy-pro.simplified-mode`. Settings and Home use the same
`useSyncExternalStore`-based helper, so changes are shared within the current tab, across tabs via
the browser storage event, and across reloads. Server rendering uses `false` as the stable fallback.

Home keeps its existing simplified/grandma experience; only its former component-only state was
replaced. The standard Home layout was not redesigned.

## 10. Agreements and beta wording

Settings includes expandable Ukrainian blocks for:

- `Умови користування`;
- `Політика конфіденційності`;
- `Правила beta-тесту`.

The copy says the product is in beta, bugs are possible, real acquiring/payments are not launched,
unnecessary sensitive data should not be shared, and final legal text requires review before a
production launch. It does not claim insurance, complete identity verification, guaranteed dispute
resolution, or production-grade legal compliance.

## 11. Problem report

The report form requires 10-1000 user-written characters and disables submission while sending. It
inserts a typed `support_requests` row with `kind = "callback"`, `duration_seconds = null`, and the
note marker `[beta_issue]` followed only by the submitted issue text.

It does not attach auth tokens, coordinates, wallet data, private addresses, chat content, device
secrets, or files. Success and failure use visible toasts.

## 12. Public/private data safety

- The Settings gear is gated by `isSelf`.
- The Settings route reads only the authenticated session owner.
- Email, role, support actions, and master debt state are not added to public Profile output.
- The master warning does not reveal wallet balance.
- Public Profile continues to load through `get_public_profile` and retains the PR #5 separation.
- Local preferences contain only language and simplified-mode values.

## 16. Remaining risks

- The remote Supabase project was not independently compared with the committed generated types in
  this PR; support submissions still depend on the deployed table and RLS matching the repository.
- `support_requests` uses human-readable note markers rather than dedicated request-kind columns.
- There is no server-side rate limit for repeated support submissions.
- Selecting Russian or English records preference only and does not translate the current UI.
- Actual deletion still needs a trusted, verified admin/server workflow with retention rules.
- Authenticated visual smoke testing requires a tester session; the unauthenticated route redirect
  to `/auth` was verified locally with no console errors.
